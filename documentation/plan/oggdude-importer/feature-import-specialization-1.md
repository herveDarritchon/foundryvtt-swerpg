# Introduction

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

Add a new domain "specialization" to the OggDude data importer. Source data resides in the OggDude ZIP under `Data/Specializations/` (multiple XML files, one per specialization, e.g. `Advisor.xml`). Each XML root `<Specialization>` contains: `Name`, `Key`, `Description` (with markup tokens like `[H4]`), `Source`, `CareerSkills` (list of skill codes), `TalentRows` (grid info referencing talent keys) and attribute blocks. Initial scope: Import Specialization as an Item of existing type `specialization` mapping description, specializationSkills (from `CareerSkills`), optional freeSkillRank (default 4), and OggDude flags. Talent tree data (`TalentRows`) is deferred (documented as future enhancement). Pattern mirrors existing `career` importer: directory aggregation, mapper producing an array of item source objects, stats tracking, global metrics integration, preload preview support, UI domain addition & localization.

## 1. Requirements & Constraints

- **REQ-001**: Load all specialization XML files from `Data/Specializations/` directory in the OggDude archive.
- **REQ-002**: Map mandatory fields: `Name`, `Key`; optional `Description`, `Source`.
- **REQ-003**: Convert OggDude markup tokens (`[H1]..[h1]`, `[B]`, `[I]`, `[UL]`, etc.) to sanitized HTML identical to career conversion logic.
- **REQ-004**: Map `CareerSkills` list (skill codes) to `system.specializationSkills` using existing `mapOggDudeSkillCodes()` logic with de-duplication and truncation to max 8 entries per data model constraint.
- **REQ-005**: Provide default `freeSkillRank = 4` if absent, bounded 0–8 (reuse career normalization function generalized).
- **REQ-006**: Flag each imported item with `flags.swerpg.oggdudeKey` and optionally `oggdudeSource` and `oggdudeSourcePage` if parsed from `<Source Page="X">`.
- **REQ-007**: Track import stats: `total`, `rejected`, `unknownSkills`, `skillCount`, `skillDetails` (Set → array) similarly to career stats.
- **REQ-008**: Increment `unknownSkills` for unmapped skill codes without halting the import; log warnings.
- **REQ-009**: Provide preload preview entries (exists flag) via `preloadOggDudeData()` for domain `specialization`.
- **REQ-010**: Integrate new domain into progress callback (processed/total) in `processOggDudeData()`.
- **REQ-011**: Update global metrics aggregator to include specialization stats in totals.
- **REQ-012**: Add localization keys for new domain selector (`SETTINGS.OggDudeDataImporter.loadWindow.domains.specialization`) in `lang/en.json` & `lang/fr.json`.
- **REQ-013**: Provide unit tests (mapper, stats utils, markup conversion, skill mapping constraints) and integration test parsing real XML files from `resources/integration/Specializations/` subset.
- **REQ-014**: Ensure description sanitization prevents injection (escape HTML entities; remove unknown tags).
- **REQ-015**: Ensure mapper returns array (filtering out nulls) compatible with `OggDudeDataElement.processElements()` API.
- **REQ-016**: Maintain Foundry VTT v13 compatibility; no usage of experimental APIs.
- **REQ-017**: Code adheres to existing importer patterns (directory-based builder analogous to career/species).
- **REQ-018**: Provide observability logs at debug level summarizing counts per specialization.
- **REQ-019**: Avoid blocking performance: parsing done with Promise.all (already in buildJsonDataFromDirectory) and lightweight per-item mapping.
- **REQ-020**: Document usage in `documentation/importer/import-specialization.md` & update importer README.
- **REQ-021**: Reuse existing `OGG_DUDE_SKILL_MAP` for specialization CareerSkills code translation; no new duplicate mapping file.
- **REQ-022**: Provide automatic coverage scan of all specialization XML `CareerSkills` codes ensuring each code either maps to a system skill id present in `SYSTEM.SKILLS` or is listed in an unknown set logged once.
- **REQ-023**: Introduce a generic skill extraction + mapping helper (`mapSpecializationSkills`) mirroring `mapCareerSkills` but decoupled from career naming for reuse.
- **REQ-024**: Add documentation section listing unmapped specialization skill codes (if any) generated by coverage test.
- **REQ-025**: Ensure specialization mapping logic gracefully handles both `<CareerSkills><Key>CODE</Key>...</CareerSkills>` and potential nested `<CareerSkill><Key>CODE</Key></CareerSkill>` variants.
- **SEC-001**: Sanitize and escape user-provided description content—no direct unsanitized insertion into HTML.
- **SEC-002**: Validate skill codes; unknown codes logged but not executed.
- **SEC-003**: Reject items with missing mandatory `Name` or `Key` (increment `rejected`).
- **CON-001**: Maximum 8 specialization skills enforced (data model constraint); truncate beyond 8.
- **CON-002**: Preserve original ordering of first occurrences; de-duplicate subsequent duplicates.
- **CON-003**: Performance: Must handle large directory (>100 files) under 2s mapping (baseline similar career importer). No synchronous heavy loops beyond linear mapping.
- **CON-004**: Memory: Avoid retaining full XML text after parsing (let it GC). Do not store large intermediate arrays besides required `jsonData`.
- **GUD-001**: Reuse existing markup conversion; extract generic functions rather than copy (self-explanatory code principle).
- **GUD-002**: Use JSDoc on public functions (mapper, context builder, util functions).
- **PAT-001**: Context builder returns `{ jsonData, zip, image, folder, element }` object; `element` includes `{ jsonCriteria, mapper, type }`.
- **PAT-002**: Stats utils pattern matches existing `career-import-utils.mjs` naming & structure.
- **PAT-003**: Domain registration pattern: add `buildContextMap.set('specialization', ...)` in both `processOggDudeData` and `preloadOggDudeData`.
- **PAT-004**: Preload preview uses same existence check logic for game items.
- **PAT-005**: Use shared markup utility for description to ensure consistency.

## 3. Alternatives

- **ALT-001**: Directly copy career mapping functions into specialization mapper (no shared util). Rejected due to duplication & maintainability (violates GUD-001).
- **ALT-002**: Parse `TalentRows` now and auto-link talents to specialization item system properties. Rejected for scope creep; requires talent key resolution & tree modeling not yet in specialization data model.
- **ALT-003**: Stream parse XML files sequentially to lower memory. Rejected; current approach already efficient and directory count moderate; premature optimization.

## 6. Testing

- **TEST-001**: Unit mapper test: maps minimal specialization object → item with `name`, `type='specialization'`, flags, description sanitized, skills truncated to <=8, duplicates removed.
- **TEST-002**: Unknown skill codes tracking: given codes `[ATHL, UNKNOWN1, ATHL, UNKNOWN2]` results: specializationSkills includes one `athletics`; stats unknownSkills=2; skillCount counts mapped skills only.
- **TEST-003**: Description markup conversion: input with `[H4]Title[h4][B]Bold[/B][I]Em[/I]` yields `<h4>` section + `<strong>` + `<em>` and sanitized closing tags; no leftover bracket tokens.
- **TEST-004**: Missing mandatory fields: object lacking `Name` or `Key` increments `rejected` and is filtered out.
- **TEST-005**: Integration test reading first N (e.g. 5) specialization XML files from `resources/integration/Specializations/` verifying array length, proper flags, skills presence, stats totals.
- **TEST-006**: Global metrics updated: after running career + specialization mapper, totalProcessed reflects sum; specialization stats accessible via `getAllImportStats().specialization`.
- **TEST-007**: Preload preview: invoking `preloadOggDudeData()` with only specialization domain returns preview objects each with `preview=true` and `exists` boolean; test with mocked `game.items`.
- **TEST-008**: FreeSkillRank normalization: invalid inputs (`"-1"`, `"20"`) clamped to range 0–8; missing defaults to 4.
- **TEST-009**: Localization presence check: domain label resolves via `game.i18n.localize` when key added (test by importing JSON and checking property).
- **TEST-010**: Skill code extraction variants: XML using direct `<CareerSkills><Key>` vs nested `<CareerSkills><CareerSkill><Key>` both produce identical raw code arrays.
- **TEST-011**: Mapping coverage: iterate all specialization XMLs; assert each code either maps via `mapOggDudeSkillCode` or appears in an `unknownCodes` set; test fails if unknown percentage > 20% (guardrail).
- **TEST-012**: Advisor specialization mapping yields specializationSkills ids `[charm, deception, negotiation, streetwise]` in any order; unknownSkills stat = 0.
- **TEST-013**: De-duplication & truncation: artificial test with >10 repeated/mixed codes results length <=8 and ordered by first occurrence.
- **TEST-014**: Unknown code logging throttled: multiple occurrences of same unknown code log only once (spy logger count assertion).
- **TEST-015**: mapSpecializationSkills strict mode: filters out ids not present in injected `SYSTEM.SKILLS` mock; preserves valid ones.

## 7. Risks & Assumptions

- **RISK-001**: Large number of XML files could slightly increase import duration; mitigated by parallel `Promise.all` already in directory builder.
- **RISK-002**: Skill codes may include ones absent from mapping causing high volume of warnings; ensure throttled logging (single warn per unknown code). (Future enhancement: aggregate unknowns.)
- **RISK-003**: TalentRows ignored may lead to incomplete specialization feature expectations for users; documented as out-of-scope to manage expectations.
- **RISK-004**: Description markup differences vs career could introduce missed patterns; tests cover representative tags.
- **RISK-005**: Duplicate names could cause user confusion (existence flag not preventing duplicates). Future improvement: de-duplication or merge.
- **ASSUMPTION-001**: Specialization item type already registered & functional (verified by presence of `module/models/specialization.mjs`).
- **ASSUMPTION-002**: XML structure for all specialization files consistent with sample (root `<Specialization>` containing similar children).
- **ASSUMPTION-003**: FreeRanks field optional; default suffices.
- **ASSUMPTION-004**: TalentRows not required for initial importer; separate feature will map talents.
